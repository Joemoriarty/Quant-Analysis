# Quant-Analysis

这是一个面向 A 股研究、候选筛选、组合约束与回测验证的一体化股票分析系统。  
系统当前采用 `Streamlit + AkShare + SQLite` 作为主底座，保留轻量部署能力，同时逐步吸收 `TradingAgents-CN` 在研究流程、执行判定、任务恢复和数据治理上的优势。

一句话定位：

- 不是单纯的“看指标网页”
- 也不是纯聊天式的“大模型选股”
- 而是把 `数据抓取 -> 统一分析 -> 候选筛选 -> 组合约束 -> 回测验证 -> 文档追踪` 串成一条完整投研链路的工具型系统

## 1. 软件想解决什么问题

传统个人量化或选股工具，常见有三个问题：

1. 单股结论和候选筛选不是同一套逻辑
2. 研究结论和交易执行之间断层明显
3. 改了一堆功能后，很难回溯“为什么这样判断、最近改了什么、下一步要补什么”

这个项目的核心思路就是把这三个问题一次性解决：

1. 用 `single_stock_analysis` 作为唯一分析内核  
   所有单股页、候选池、组合选股、回测调仓都尽量复用同一套分析口径
2. 用结构化研究层和执行层把“研究”往“交易动作”下沉  
   不只给推荐结论，还给目标价情景、执行置信度、执行风险分、风险修正动作
3. 用 `docs/` 体系把逻辑、架构、专业化差距、外部项目借鉴路线和历史变更全部记录下来  
   方便产品迭代、对外汇报和后续接手

## 2. 软件的核心逻辑与思路

### 2.1 一条主链，不维护多套口径

系统的核心判断主链是：

`single_stock_analysis -> candidate_screener -> unified_selection -> portfolio_backtester`

这条主链的意义是：

1. 先把单只股票分析清楚
2. 再把同一套结果复用到候选池
3. 再把候选池结果复用到组合层
4. 再把组合层决策复用到回测层

这样做的好处是：

- 单股页看到的判断，不会和候选页完全两套说法
- 回测里出现的选股，也能回到单股分析里解释原因
- 页面、策略、回测之间的口径更容易统一

### 2.2 六层评估，不只看技术面

当前系统不是只靠均线或量价信号做判断，而是把一只股票拆成多个维度评估：

1. 技术面  
   解决“趋势是否成立、位置是否合适、短期是否过热”
2. 基本面  
   解决“财务质量、增长、估值是否支持当前结论”
3. 市场情绪  
   解决“当前大盘环境是否顺风、是否逆风”
4. 事件驱动  
   解决“公告、业绩预告、披露节奏是否构成额外催化或风险”
5. 行业横向比较  
   解决“放到同行里，这只股票到底处于什么位置”
6. 执行层  
   解决“即便研究结论成立，现在是否值得做、怎么做、风险有多大”

这六层最终会沉淀为几个对外最关键的输出：

1. 最终推荐结论
2. 最终结论依据
3. 研究流程摘要
4. 风险委员会摘要
5. 目标价情景
6. 执行计划摘要

### 2.3 研究和交易执行分开

这是系统当前很重要的一条产品思路。

很多工具做到“推荐关注”就结束了，但真实投资里还需要回答：

1. 现在应该买还是等
2. 应该轻仓试探还是正常配置
3. 如果看错了，什么条件下撤退
4. 如果判断继续强化，什么条件下升级

所以系统除了研究结论，还会补充：

1. `research_workflow_summary`  
   用结构化方式输出投资逻辑、看多依据、反方依据、跟踪指标、失效条件
2. `risk_committee_summary`  
   用结构化方式输出趋势风险、基本面风险、事件风险、行业/组合风险
3. `target_price_scenarios`  
   输出保守、基准、乐观三档目标价情景
4. `execution_plan_summary`  
   输出风险修正动作、建议目标仓位、执行置信度、执行风险分、止损止盈参考

这一步让系统从“研究展示工具”更接近“研究到执行”的决策工具。

## 3. 软件执行流程

如果从用户使用视角来看，当前软件的完整执行流程可以概括为下面 7 步：

### 第 1 步：加载股票池与基础数据

系统先加载股票列表、基础行情、缓存快照和数据库中已有记录。  
为了兼顾部署速度和外部接口稳定性，当前使用：

- `AkShare` 作为主要事实数据源
- `SQLite` 作为本地持久化存储
- `.cache` 作为页面热数据缓存
- `Tushare` 作为部分基本面字段备源

### 第 2 步：对单只股票做统一分析

进入单股分析后，系统会围绕这只股票依次完成：

1. 技术面打分与趋势判断
2. 基本面快照与质量校正
3. 市场情绪环境校正
4. 事件驱动摘要
5. 行业横向比较与行业热度判断
6. 实时新闻补充解释

这一步的目标是产出“唯一标准分析结果”。

### 第 3 步：生成结构化研究结论

单股分析结果不是直接停在分数上，而是继续组织成：

1. 投资逻辑
2. 看多依据
3. 看空依据
4. 跟踪指标
5. 失效条件

这样更方便研究复盘、投研讨论和对外汇报。

### 第 4 步：生成风险裁决与执行计划

在研究结论基础上，系统会继续判断：

1. 当前风险属于什么等级
2. 是否需要把动作从“买入”下调为“观察”
3. 建议目标仓位是多少
4. 目标价区间和止损止盈怎么参考

这一步的作用是把研究层真正下沉成交易层。

### 第 5 步：候选池筛选

量价代理候选、中线候选等入口不再单独维护旧逻辑，而是复用统一分析结果，综合考虑：

1. 趋势分
2. 基本面评分
3. 市场情绪
4. 事件驱动
5. 行业横向比较
6. 执行置信度
7. 执行风险分

也就是说，候选池已经不只是“谁分高”，而是“谁更值得现在做”。

### 第 6 步：组合层风险约束

进入组合推荐或回测之前，系统会做基础组合约束，当前已经接入：

1. 最大单票权重
2. 单行业最多持仓数
3. 最低成交额估算
4. 最低执行置信度
5. 最高执行风险分

这一步的目标不是做成完整机构风控系统，而是先避免“研究结论不错，但组合不可落地”的情况。

### 第 7 步：回测验证与记录留痕

策略执行结果会进入回测与数据库留痕，帮助回答：

1. 当时为什么选这只股票
2. 当时为什么是这个动作
3. 哪些约束导致某些股票被过滤
4. 这套执行层是否真的改善了结果

这一步让系统具备从研究到复盘的闭环。

## 4. 软件当前的主要优势

下面这部分可以直接作为 PPT 里“系统优势”页的素材。

### 4.1 统一分析内核

- 单股分析、候选筛选、组合推荐、回测调仓尽量共用一条主链
- 减少“页面上这么说，回测里又是另一套”的割裂感

### 4.2 不是单一指标，而是多维交叉验证

- 技术、基本面、情绪、事件、行业、执行 六层共同参与判断
- 比单纯技术指标工具更接近真实投研思路

### 4.3 从研究结论走到执行计划

- 不只输出“推荐/不推荐”
- 还能输出目标价情景、仓位建议、风险修正动作和执行风险

### 4.4 兼顾轻量部署与可追踪迭代

- 用 `Streamlit + SQLite` 保持低门槛部署
- 用 `docs/` 体系记录逻辑、架构、缺陷、计划和历史变更
- 既能快速迭代，也便于长期维护

### 4.5 对外部数据源更稳健

- 主源失败时尽量走缓存和备源
- 实时新闻、基本面备源、行业热度都在逐步补齐
- 比完全依赖单次在线请求更稳

### 4.6 借鉴开源框架，但不照搬重型架构

- 吸收 `TradingAgents-CN` 的研究流程、风险裁决、执行层和任务恢复思想
- 不直接迁移到重型多智能体平台
- 更适合当前单人/小团队持续迭代

## 7. 启动方式

请优先使用项目虚拟环境，不要直接使用系统 `python3`。

```bash
.venv/bin/python run.py
```

Render 部署建议直接使用：

```bash
streamlit run web/app.py --server.port $PORT --server.address 0.0.0.0 --server.headless true --browser.gatherUsageStats false
```

不要把 `run.py` 再包一层 `streamlit run`，也不要填写不存在的 `web/run.py`。仓库已提供 `render.yaml`，可以直接按 Blueprint 导入。

如果希望缓存、数据库、观察列表、模拟交易记录在 Render 重启后继续保留，请挂载 Persistent Disk，并把 `APP_STORAGE_DIR` 指向持久盘目录。当前仓库的 Blueprint 默认使用：

```text
APP_STORAGE_DIR=/var/data/quant-analysis
```

启用后，下列内容都会持久化到该目录下：

- `db/market_data.db`
- `data/cache/*.csv`
- `data/watchlist/watchlist.csv`
- `data/paper_trading/*`

## 8. 自动任务脚本

如果你想从命令行或外部调度器触发任务，可以使用：

```bash
.venv/bin/python automation_runner.py daily-update --pool-size 300
.venv/bin/python automation_runner.py weekly-optimize --pool-size 300
```

这两个命令会分别执行：

- 每日更新：拉取更大股票池、同步行情到数据库
- 每周优化：基于当前股票池重新评估参数，并把最优参数写回数据库

## 9. 常见问题

### 9.1 为什么部署后首次打开偏慢

当前部署在 Render 免费版时，服务休眠后首次访问会有冷启动。  
项目已经通过缓存、数据库快照和后台刷新尽量降低重复计算，但无法完全避免免费环境的首开延迟。

### 9.2 为什么有些行业比较或新闻会不稳定

根因通常不是本地逻辑错误，而是外部财经数据源可用性波动。  
当前系统已经尽量加入：

- 本地缓存
- 数据库快照
- 主备源切换

但外部源不可达时，部分结果仍可能降级或缺失。

### 9.3 这是不是完整的机构级投研平台

还不是。  
当前系统已经具备比较清晰的投研主链和第一版组合约束，但在相关性矩阵、风格暴露、Beta 暴露、完整回撤控制等方面，仍在继续增强。
